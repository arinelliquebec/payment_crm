# `seed/` — Synthetic demo data

This folder contains SQL seeds for the public portfolio demo of the Payment
CRM project. Everything here is **100% fictional**, generated from algorithms
and neutral catalogs. It contains **no** data sourced from any real customer,
partner, ex-employer system, or third-party dataset.

See [`docs/data-policy.md`](../docs/data-policy.md) for the project-wide policy
on synthetic data; this README documents the operational procedure.

---

## Generation strategy

* **Names**: prefixed with `[DEMO]` and a sequential index (`Pessoa Demo 001`,
  `Empresa Exemplo 02 Ltda`). No real person or company name is used anywhere.
* **CPF / CNPJ**: generated by Brazilian module-11 algorithm inside the SQL
  file (`pg_temp.cpf_format`, `pg_temp.cnpj_format`). All CPFs start with
  prefix `999`; all CNPJs start with prefix `99`. Values are checksum-valid
  but the prefixes flag them as demo-only.
* **Emails**: only reserved domains `example.com`, `example.org`,
  `example.net` (RFC 2606) and `demo.local`, `test.local` (private-use).
* **Phones**: only reserved test prefixes `(11) 90000-XXXX` (mobile) and
  `(11) 3000-XXXX` (landline). No real number falls in these ranges in
  combination with the `[DEMO]` markers.
* **CEP**: starts with `9` to reduce overlap with realistic Brazilian ranges.
* **Money**: derived from deterministic arithmetic on the row index; ranges
  R$ 100 – R$ 50 000.
* **Dates**: relative to `now()`, so dashboards and timelines are not frozen
  in the past.
* **Ids**: explicit, all starting at `10_000_000`, in disjoint slabs per
  table, to avoid colliding with rows produced by the application's own ID
  generator. See [ID strategy](#id-strategy) below.
* **Passwords**: stored as the literal placeholder
  `__DEMO_PLACEHOLDER_RESET_BEFORE_USE__`. **Never functional.** Login is
  impossible until passwords are reset through the application — see
  [Setting passwords](#setting-passwords-after-seed) below.

## Safety properties of `seed-demo.sql`

1. **Single transaction.** Any error rolls back everything.
2. **Pre-flight gate.** Refuses to run if the target schema already contains
   `PessoasFisicas` or `PessoasJuridicas` rows whose names do not start with
   `[DEMO]`. This protects against accidentally mixing demo rows with real
   data that may have ended up in the dev DB. To override (e.g. you know the
   coexistence is intentional), run **before** the seed:
   ```sql
   SET demo.allow_non_empty TO 'true';
   ```
3. **Idempotent.** Re-running is safe. Each table relies on `ON CONFLICT DO
   NOTHING` plus the explicit `Id` strategy, so a second run is a no-op.
4. **No DDL on application tables.** The only `CREATE` is for two helper
   functions in `pg_temp` (auto-dropped at session end).
5. **No real credentials.** No bcrypt hashes, API keys, tokens, or secrets
   anywhere in the file.
6. **No data destruction.** No `DELETE` / `UPDATE` / `TRUNCATE` / `DROP`.

## How to run (only after explicit approval)

```bash
# 1. Use a connection string for the DEV database. Read it from your local
#    secrets manager / 1Password / Azure Key Vault. Do NOT commit it.
export DEV_DB_URL='postgresql://user:pass@host:5432/dbname?sslmode=require'

# 2. Dry-validate the file (parse + transaction begin/rollback) without
#    persisting:
psql "$DEV_DB_URL" -v ON_ERROR_STOP=1 \
  -c 'BEGIN;' \
  -f seed/seed-demo.sql \
  -c 'ROLLBACK;'

# 3. Real run (commits):
psql "$DEV_DB_URL" -v ON_ERROR_STOP=1 -f seed/seed-demo.sql

# 4. Verify counts:
psql "$DEV_DB_URL" -c '
  SELECT ''Filiais'' AS t, count(*) FROM "Filiais" WHERE "Nome" LIKE ''[DEMO]%''
  UNION ALL SELECT ''PessoasFisicas'',  count(*) FROM "PessoasFisicas"  WHERE "Nome" LIKE ''[DEMO]%''
  UNION ALL SELECT ''PessoasJuridicas'',count(*) FROM "PessoasJuridicas" WHERE "RazaoSocial" LIKE ''[DEMO]%''
  UNION ALL SELECT ''Usuarios'',        count(*) FROM "Usuarios" WHERE "Login" LIKE ''demo-%''
  UNION ALL SELECT ''Clientes'',        count(*) FROM "Clientes" WHERE "Observacoes" LIKE ''[DEMO]%''
  UNION ALL SELECT ''Contratos'',       count(*) FROM "Contratos" WHERE "NumeroPasta" LIKE ''[DEMO]-%'';
'
```

Step 2 is a **safe dry-run**: the explicit `BEGIN ... ROLLBACK` wrapper
prevents any of the inserts from being committed even though the script
itself has its own internal transaction.

## Setting passwords after seed

The seed inserts the literal string `__DEMO_PLACEHOLDER_RESET_BEFORE_USE__`
into `Usuarios.Senha`. This is intentional: the placeholder is not a valid
bcrypt hash, so no demo user can authenticate until passwords are explicitly
set.

Two recommended ways to set them, **without committing real hashes to git**:

### Option A — Use the application's password-reset flow

Trigger the "Esqueci a senha" / `POST /api/Auth/ForgotPassword` endpoint for
each demo user (login `demo-admin`, `demo-gestor`, …). The application will
generate a token, hash the new password, and update the row in-place.

### Option B — Update via psql with a hash from your secrets manager

```bash
# Read the bcrypt hash from your secrets manager — never store it in the repo.
HASH="$(op read 'op://Personal/payment-crm-demo/users-bcrypt-hash')"

psql "$DEV_DB_URL" -v ON_ERROR_STOP=1 -c "
  UPDATE \"Usuarios\"
     SET \"Senha\" = '$HASH'
   WHERE \"Login\" LIKE 'demo-%'
     AND \"Senha\" = '__DEMO_PLACEHOLDER_RESET_BEFORE_USE__';
"
```

The `WHERE … = '__DEMO_PLACEHOLDER_RESET_BEFORE_USE__'` clause guarantees the
update affects only freshly-seeded rows, never users that already had real
passwords.

## ID strategy

The dev schema uses `integer` PKs without a visible default or sequence. The
application generates ids client-side. To avoid colliding with rows the app
will produce later, the seed uses high explicit ids:

| Table                          | Range                       |
| ------------------------------ | --------------------------- |
| `Filiais`                      | `10000001 .. 10000003`      |
| `Enderecos`                    | `10100001 .. 10100055`      |
| `PessoasFisicas`               | `10200001 .. 10200040`      |
| `PessoasJuridicas`             | `10300001 .. 10300015`      |
| `Consultores`                  | `10400001 .. 10400005`      |
| `Parceiros`                    | `10500001 .. 10500003`      |
| `Usuarios`                     | `10600001 .. 10600006`      |
| `Clientes`                     | `10700001 .. 10700030`      |
| `HistoricoConsultores`         | `10800001 .. 10800030`      |
| `Contratos`                    | `10900001 .. 10900060`      |
| `HistoricoSituacaoContratos`   | `11000001 .. 11000060`      |

If your DB later turns out to use `GENERATED ALWAYS AS IDENTITY` columns
(unlikely given current `default = NULL`), the inserts will fail with a clear
error. In that case, prepend `OVERRIDING SYSTEM VALUE` to each `INSERT` or
switch the column to `GENERATED BY DEFAULT AS IDENTITY` via a proper EF Core
migration (not handled by this seed).

## Purge (removing the demo rows)

There is no purge script in this folder by design — deletion of demo data
should be a deliberate, reviewed action. When you do need it, the markers
make targeting unambiguous. Suggested ordering, respecting `ON DELETE
CASCADE` on `HistoricoConsultores` (→ `Clientes`) and
`HistoricoSituacaoContratos` (→ `Contratos`):

```sql
BEGIN;

DELETE FROM "Contratos"
 WHERE "NumeroPasta" LIKE '[DEMO]-%';
-- HistoricoSituacaoContratos rows are removed by ON DELETE CASCADE.

DELETE FROM "Clientes"
 WHERE "Observacoes" LIKE '[DEMO]%';
-- HistoricoConsultores rows are removed by ON DELETE CASCADE.

DELETE FROM "Usuarios"
 WHERE "Login" LIKE 'demo-%';

DELETE FROM "Parceiros"
 WHERE "OAB" LIKE '[DEMO]%';

DELETE FROM "Consultores"
 WHERE "OAB" LIKE '[DEMO]%';

DELETE FROM "PessoasJuridicas"
 WHERE "RazaoSocial" LIKE '[DEMO]%';
-- The matching Enderecos rows go via ON DELETE CASCADE on
-- PessoasJuridicas.EnderecoId.

DELETE FROM "PessoasFisicas"
 WHERE "Nome" LIKE '[DEMO]%';
-- Enderecos for PessoasFisicas use ON DELETE SET NULL, so delete them
-- explicitly:

DELETE FROM "Enderecos"
 WHERE "Logradouro" LIKE '[DEMO]%';

DELETE FROM "Filiais"
 WHERE "Nome" LIKE '[DEMO]%';

COMMIT;
```

Run on dev only, never on production.
